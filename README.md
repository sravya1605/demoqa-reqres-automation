# DemoQA + ReqRes Test Automation

![Playwright Tests](https://github.com/sravya1605/demoqa-reqres-automation/actions/workflows/playwright.yml/badge.svg)

End-to-end and API test automation built with **Playwright + JavaScript**, covering:

- **UI**: manual-user login, Book Store search, result validation, CSV export, and logout on [demoqa.com](https://demoqa.com)
- **API**: create / fetch / update user flow (plus a negative-path check) against [reqres.in](https://reqres.in)

Built as a QA Automation assignment. The goal was a small, correct, maintainable framework — not an over-engineered one — with every locator and API behavior verified against the live sites rather than assumed from documentation or tutorials.

## Test Report

A run of this suite currently looks like this:

| Passed | Failed | Skipped | Flaky |
| ------ | ------ | ------- | ----- |
| 5      | 0      | 0       | 0     |

**To generate and view the report yourself:**

1. `npm test` — runs the full suite and writes a report to `playwright-report/`
2. `npm run report` — opens that report in your browser
3. The page that opens is the **test run summary** (a list of every test, grouped by browser project, each with a green check or red X)
4. Click into any test to expand its `test.step()` breakdown, and — for the API test specifically — the actual request/response bodies

**What to screenshot:** the top-level summary page (`npm run report`, the first screen that opens) showing all tests green, is the one screenshot that answers "did it pass" at a glance. If you want a second, more detailed screenshot, expand one test to show its step-by-step breakdown.

**Where to share it:** attach the screenshot directly to the submission email, since that's literally what was requested. It's also worth linking the GitHub Actions run itself (Actions tab → latest green run) — that's a live, permanent, always-up-to-date version of the same proof, and every CI run also posts a plain-text pass/fail summary directly on the run page itself (no download needed) via a small script (see CI Pipeline below).

## Tech Stack

| Tool                                               | Purpose                                                  |
| --------------------------------------------------- | ----------------------------------------------------------- |
| Playwright Test                                    | Test runner, browser automation, API testing, reporting |
| ESLint (flat config) + `eslint-plugin-playwright`  | Linting                                                  |
| Prettier                                           | Code formatting                                          |
| dotenv                                             | Environment configuration                                |

No custom API client, CSV library, or wait-utility packages were added — Playwright's `request` fixture and Node's built-in `fs` module already cover everything this project needs.

## Setup

```bash
npm install
npx playwright install --with-deps
cp .env.example .env
```

Fill in `.env`:

```
UI_BASE_URL=https://demoqa.com
API_BASE_URL=https://reqres.in
DEMOQA_USERNAME=<a user you create manually>
DEMOQA_PASSWORD=<that user's password>
REQRES_API_KEY=<a free key from https://app.reqres.in>
```

> Registration is intentionally **not** automated, per the assignment. Create a user yourself at [demoqa.com/register](https://demoqa.com/register) first, then put its credentials in `.env`.

> reqres.in's `/api/users` endpoints require a free API key as of 2026 (their own public docs describing these as unauthenticated are out of date). Sign up at [app.reqres.in](https://app.reqres.in) - no card required - and copy the key shown on your dashboard.

## CI Pipeline

Every push triggers `.github/workflows/playwright.yml`, which:

1. Installs dependencies and Playwright's browsers
2. Runs the full suite (`npm test`)
3. Posts a plain pass/fail summary directly to the Actions run page (`scripts/post-ci-summary.js` reads Playwright's JSON reporter output and writes it to the GitHub step summary) — so results are visible at a glance without downloading anything
4. Uploads the full HTML report as a downloadable artifact, on every run, pass or fail

## Known Issues & Limitations

- **reqres.in does not persist demo data.** `POST /api/users` returns a freshly generated `id`, but that record never actually exists server-side afterward — `GET /api/users/:id` only returns real data for reqres.in's 12 seeded fixture users (ids 1–12). This project's "Fetch Existing User" step deliberately validates a known seeded user (`id: 2`, `janet.weaver@reqres.in`) instead of the just-created id, since asserting persistence the API doesn't provide would be testing a false assumption, not the real API contract. The "Update User" step does use the real captured id, since `PUT` doesn't require the id to correspond to an existing record on this endpoint.
- **reqres.in requires an API key.** Confirmed directly against the live API from multiple networks (`401 missing_api_key` every time) — their own public documentation (`reqres.in/llm.txt`) still describes `/api/users` as unauthenticated; that documentation is out of date. A free key from [app.reqres.in](https://app.reqres.in) resolves this.
- **demoqa.com ad overlays.** The site occasionally serves ad content that can visually overlap page elements. No `force: true` or artificial waits were added to mask this — if it happens, Playwright's own actionability timeout surfaces it clearly as a failed step, which is the correct behavior rather than silently clicking through a covered element.
- **demoqa.com markup has changed from older references.** All locators here were confirmed against the live site directly (not assumed from tutorials or older documentation). Specifics worth knowing if this ever needs updating again: the login username field's placeholder is `UserName` with no space, the Book Store nav button reads "Go To Book Store," the logout button's label is inconsistent between pages (`Logout` on the profile page, `Log out` on the Book Store page — handled with a case-insensitive regex), and the results table is a plain semantic `<table>` rather than the `react-table` library older demoqa references assume.

## Assumptions

- **Manual user creation**: per the assignment, registration is not automated. A user must be created once, manually, and its credentials placed in `.env`.
- **Search result assertions**: the "Validate Search Result" step asserts the returned title matches the search term and that author/publisher are non-empty, rather than hardcoding exact author/publisher text — this keeps the test resilient to minor catalog data changes while still catching a broken extraction.

## CSV Output

The UI test writes the searched book's details to `output/book-details.csv` (columns: `Title, Author, Publisher`), overwriting on each run so results don't accumulate across executions.

## Future Improvements

- Reuse a saved `storageState` in `playwright/.auth/` to skip the login UI step for tests that don't need to validate login itself.
- Add a UI negative-path test (e.g., invalid login credentials, searching for a book that doesn't exist) to mirror the API suite's negative-path coverage.
- Parameterize the searched book title via `.env` or a test fixture if more book-search scenarios are added later.
- Add a `data-testid`-based locator strategy if demoqa.com ever exposes one, as a more resilient alternative to text/placeholder locators for elements without accessible roles.
