# DemoQA + ReqRes Test Automation

Playwright (JavaScript) test automation covering:

- **UI**: login, Book Store search, CSV export, and logout on [demoqa.com](https://demoqa.com)
- **API**: create / fetch / update user flow against [reqres.in](https://reqres.in)

## Tech Stack

| Tool                                              | Purpose                                                 |
| ------------------------------------------------- | ------------------------------------------------------- |
| Playwright Test                                   | Test runner, browser automation, API testing, reporting |
| ESLint (flat config) + `eslint-plugin-playwright` | Linting                                                 |
| Prettier                                          | Code formatting                                         |
| dotenv                                            | Environment configuration                               |

No custom API client, CSV library, or wait-utility packages were added — Playwright's `request` fixture and Node's built-in `fs` module already cover everything this project needs.

## Folder Structure

```
├── api/                 # Plain functions wrapping Playwright's request fixture (reqres.in endpoints)
├── config/              # Centralized environment configuration (config/env.js)
├── fixtures/            # Custom Playwright test fixture (injects page objects into tests)
├── pages/               # Page objects (BasePage, LoginPage, BookStorePage)
├── utils/               # Small, single-purpose helpers (CSV writer, test data generator)
├── tests/
│   ├── ui/               # demoqa.com Book Store flow
│   └── api/              # reqres.in user CRUD flow
├── output/               # book-details.csv (generated by the UI test)
├── playwright/.auth/     # Reserved for storageState reuse (see its own README)
├── playwright.config.js
└── eslint.config.js
```

## Setup

```bash
npm install
npx playwright install --with-deps
cp .env.example .env
```

Fill in `.env`:

```
UI_BASE_URL=https://demoqa.com
API_BASE_URL=https://reqres.in
DEMOQA_USERNAME=<a user you create manually>
DEMOQA_PASSWORD=<that user's password>
```

> Registration is intentionally **not** automated, per the assignment. Create a user yourself at [demoqa.com/register](https://demoqa.com/register) first, then put its credentials in `.env`.

## Running Tests

```bash
npm test              # everything, all three browsers
npm run test:ui       # UI (Book Store) flow only
npm run test:api      # API (reqres.in) flow only
npm run test:chromium # single browser
npm run test:firefox
npm run test:webkit
npm run test:headed   # watch the browser while it runs
npm run test:debug    # Playwright Inspector
```

## Viewing the Report

```bash
npm run report
```

Opens the HTML report generated by the last run, including step-by-step `test.step()` breakdowns, and — on any failed test — a screenshot, video, and trace.

## Known API Limitation: reqres.in Does Not Persist Demo Data

reqres.in's free `/api/users` endpoints **echo back whatever you send but never actually store it** (confirmed in reqres.in's own documentation). This has a direct consequence for the assignment's "create → fetch → update" flow:

- `POST /api/users` returns a freshly generated `id` in the response — but that record doesn't exist server-side afterward.
- `GET /api/users/:id` only returns real data for reqres.in's 12 seeded fixture users (ids 1–12). Calling it with the id you just "created" would return `404`, not because anything is broken, but because that's the documented behavior of the endpoint.
- `PUT /api/users/:id` always returns `200` and echoes the update, regardless of whether the id refers to a real record.

**How this project handles it:** the "Fetch Existing User" step validates `GET /api/users/2` (a known seeded fixture, `janet.weaver@reqres.in`), while "Update User" uses the id captured from the "Create User" step. This tests the real, documented contract of the API rather than asserting on behavior the API doesn't actually provide. See the comment at the top of `tests/api/users.spec.js` for the same explanation in context.

## Assumptions Made for DemoQA

- **Manual user creation**: per the assignment, registration is not automated. A user must be created once, manually, and its credentials placed in `.env`.
- **Search result assertions**: the "Validate Search Result" step asserts the returned title matches the search term and that author/publisher are non-empty, rather than hardcoding exact author/publisher text — this keeps the test resilient to minor catalog data changes while still catching a broken extraction.
- **Ad overlays**: demoqa.com is known to occasionally serve ad content that can visually overlap page elements. No `force: true` or artificial waits were added to mask this — if it happens, Playwright's own actionability timeout will surface it clearly as a failed step, which is the correct behavior rather than clicking through a covered element.
- **Locator confidence**: most locators (login fields, buttons, `#searchBox`, `#userName-value`) are long-standing, well-documented demoqa.com conventions. The Book Store results table (`.rt-tbody .rt-tr-group`, `.rt-td`) is generated by the `react-table` library demoqa uses internally — stable, but worth re-checking if demoqa ever changes that dependency.

## CSV Output

The UI test writes the searched book's details to `output/book-details.csv` (columns: `Title, Author, Publisher`), overwriting on each run so results don't accumulate across executions.

## Future Improvements

- Reuse a saved `storageState` in `playwright/.auth/` to skip the login UI step for tests that don't need to validate login itself.
- Add a negative-path API test (e.g., updating a non-existent id, or malformed payloads) if the assignment scope expands.
- Parameterize the searched book title via `.env` or a test fixture if more book-search scenarios are added later.
